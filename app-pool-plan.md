# 测试应用池构建方案 v3.1（pm-cockpit 精细开发版）

> 版本：v3.1 | 日期：2026-04-10
> 定位：**pm-cockpit skill 流程** — PRD 驱动 + Fabrice 开发 + Moss 验收
> 需求方：廖雨亭 | 方案设计：Pichai（PM）
> 变更：v3.1 — 新增智子应用类型标签；v3.0 — 结合智子 mano-afk 适配性分析，调整优先级 + 新增开发适配性标注 + 补充/替换应用

---

## 设计原则

1. **场景优先** — 从真实业务场景出发设计应用，组件是场景的自然产物
2. **双维度适配标注** — 同时标注 mano-cua 验收适配性和 mano-afk 开发适配性
3. **优先级排序** — 综合考虑验收可行性、开发适配性、组件覆盖增量
4. **覆盖率审计** — 设计完所有应用后，统一审计组件覆盖情况

---

## 标注说明

**mano-cua 验收适配性：**
- 🟢 完全适配
- 🟡 基本适配，部分交互需策略
- 🔴 含难验收组件（拖拽/文件对话框/精确取色等）

**mano-afk 开发适配性：**
- ✅ 完美 — LLM 常识完全覆盖，一句话需求即可
- ⚠️ 可做 — 主流程没问题，边界场景需 PRD 约束
- ❌ 不适合 — 必须精确 PRD + pm-cockpit 流程

### Config Provider 说明

Config Provider 是全局配置组件（国际化、主题），不属于某个应用的 UI 组件，而是作为应用根组件包裹使用。在 App 05（系统设置中心）中自然使用——语言/主题切换就是通过 Config Provider 实现的。

### mano-cua 能力边界组件汇总

| 组件 | 所在应用 | 挑战点 | 预期影响 |
|------|---------|--------|---------|
| Upload | App 15 文件管理器 | 触发系统文件对话框，mano-cua 无法控制 OS 级 UI | 上传相关用例预期 BLOCKED |
| Slider | App 15、App 16 | 拖拽操作，需精确定位 | 可能部分 PASS（点击方式），部分 BLOCKED（拖拽方式） |

| Splitter | App 16 主页生成器 | 面板边界拖拽调节 | 预期 BLOCKED |
