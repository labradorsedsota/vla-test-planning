# 测试应用分类体系

> 版本：v1.0 | 日期：2026-04-10
> 基于 VLA Phase 0 实践经验 + mano-cua 工具能力边界

---

## 一、分类逻辑

按**应用复杂度**（状态管理深度 × 交互复杂度 × 外部依赖）分为 A/B/C/D 四级。

核心目的：为 VLA 模型训练数据采集定义应用的复杂度梯度，从简单到复杂逐步验证 mano-cua 的自动化验收能力。

---

## 二、四级分类定义

### A 类 — 纯工具型（无状态）

| 维度 | 描述 |
|------|------|
| 核心特征 | 输入 → 处理 → 可验证输出，无持久化 |
| 技术栈 | 纯前端，单页面，无后端 |
| 页面数 | 1 |
| 交互复杂度 | 低（表单输入、按钮触发、结果展示） |
| 验收难度 | 低（输出可直接文本/视觉比对） |
| 验收用例量 | 15-20 条 |
| mano-cua 适配 | ✅ 完全适配 |
| 典型例子 | Markdown 转换器、JSON Formatter、单位换算器、颜色选择器 |
| Phase 0 实例 | md2wechat ✅ |

### B 类 — 状态管理型（有 CRUD）

| 维度 | 描述 |
|------|------|
| 核心特征 | 有数据持久化，多页面，含增删改查 |
| 技术栈 | 前端 + 后端 + 数据库（如 Flask+SQLite、Vue+Express） |
| 页面数 | 2-5 |
| 交互复杂度 | 中（表单提交、列表操作、页面跳转、数据联动） |
| 验收难度 | 中（需验证数据一致性、状态流转、页面间联动） |
| 验收用例量 | 20-30 条 |
| mano-cua 适配 | ✅ 基本适配 |
| 典型例子 | 记账本、待办清单、通讯录管理、个人博客后台 |
| Phase 0 实例 | tripsplit（实际已达到 B 类复杂度） |

### C 类 — 业务流程型（含角色/权限/流程）

| 维度 | 描述 |
|------|------|
| 核心特征 | 多角色、权限控制、业务流程有先后依赖 |
| 技术栈 | 前后端分离 + 认证体系 + 复杂业务逻辑 |
| 页面数 | 5+ |
| 交互复杂度 | 高（登录/注册、角色切换、审批流、状态机） |
| 验收难度 | 高（需模拟多角色操作、验证权限隔离、流程完整性） |
| 验收用例量 | 30-50 条 |
| mano-cua 适配 | ⚠️ 有挑战（多账号切换、登录态管理、流程依赖顺序） |
| 典型例子 | 订单管理系统（OMS）、审批系统、CMS、工单系统 |

### D 类 — 复杂集成型（含外部依赖/实时交互）

| 维度 | 描述 |
|------|------|
| 核心特征 | 依赖外部 API/服务、实时通信、复杂前端交互 |
| 技术栈 | 微服务/第三方 API 集成 + WebSocket/SSE + 复杂 UI 组件 |
| 页面数 | 5+ |
| 交互复杂度 | 极高（拖拽、实时协作、地图、图表、富文本编辑） |
| 验收难度 | 极高（外部依赖不可控、时序敏感、UI 交互非标准化） |
| 验收用例量 | 50+ 条 |
| mano-cua 适配 | ❌ 部分不可行（拖拽不支持、实时数据难断言） |
| 典型例子 | 在线文档协作、数据看板、地图应用、聊天应用 |

## 三、对比速查

| 维度 | A 类 | B 类 | C 类 | D 类 |
|------|------|------|------|------|
| 状态管理 | 无 | 有 | 有+角色 | 有+外部 |
| 页面数 | 1 | 2-5 | 5+ | 5+ |
| 数据持久化 | 无 | 本地/DB | DB+认证 | DB+第三方 |
| 验收用例量 | 15-20 | 20-30 | 30-50 | 50+ |
| mano-cua 适配 | ✅ 完全 | ✅ 基本 | ⚠️ 有挑战 | ❌ 部分不可行 |

## 四、验收层级（L1/L2/L3）

每个应用内部的测试用例按层级分类：

| 层级 | 定义 | 通过标准 |
|------|------|---------|
| L1 | 基础功能 — 核心 happy path 必须通过 | 100% |
| L2 | 交互体验 + 边界情况 | 100% |
| L3 | 高级特性 + 极端场景 + 鲁棒性 | 95% |

---

## 五、版本分类（Golden / Buggy）

| 版本 | 定义 | 用途 |
|------|------|------|
| Golden | 功能完全正确的应用版本 | 正样本来源 |
| Buggy | 在 Golden 基础上植入 2-3 个真实感 Bug | 负样本来源 |

**Buggy 版本要求：**
- Bug 由开发自主设计（非 PM 指定），更接近真实开发疏忽
- Bug 应覆盖不同类型（逻辑错误、数据错误、UI 错误等）
- Bug 有真实感，不能是明显的刻意破坏
